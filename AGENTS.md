# AGENTS.md

이 문서는 `Baby Dragon Incremental` 프로젝트에서 작업하는 에이전트/개발자를 위한 공통 지침서다.

## 1) 프로젝트 목적
- 장르: 2D 픽셀 싱글플레이 인크리멘탈.
- 핵심 판타지: 먹어치우며 성장하는 아기 용.
- 핵심 진행: 단일 거대 스킬트리 + 상위 맵 해금 + 최종 보스.
- 엔딩 조건: 모든 스킬트리 해금, 모든 맵 해금, 최종 보스 처치.

## 2) 기술/범위 제약
- 엔진: LÖVE2D (Lua).
- 플랫폼: PC 단일.
- 네트워크/서버: 사용하지 않음.
- 저장: 로컬 파일 저장만 사용 (`save.json`, `save.bak`).
- 오프라인 보상/환생(프레스티지): v1 범위에서 제외.

## 3) 소프트웨어 설계 정론 (필수)
- `SOLID`를 모듈 단위로 적용한다.
- `S` 단일 책임 원칙:
  - 모듈은 하나의 이유로만 변경되어야 한다.
  - 예: `food_system`은 스폰/섭취 계산만 담당하고 저장/렌더 책임을 갖지 않는다.
- `O` 개방-폐쇄 원칙:
  - 새 기능은 기존 모듈 내부 분기 추가보다 신규 모듈/데이터 추가로 확장한다.
  - 예: 맵 추가는 `constants` 데이터 확장으로 해결하고 코어 루프 수정은 최소화한다.
- `L` 리스코프 치환 원칙(동등 계약):
  - 같은 역할의 구현은 동일한 입력/출력 계약을 지켜 교체 가능해야 한다.
  - 예: 저장 구현 교체 시 `load/save` 성공/실패 계약은 유지한다.
- `I` 인터페이스 분리 원칙:
  - 큰 범용 API 하나보다 작은 목적형 함수로 분리한다.
  - 예: `saveNow`, `loadOrDefault`, `snapshot`처럼 목적별 API 유지.
- `D` 의존성 역전 원칙:
  - 상위 정책은 하위 세부사항에 직접 결합하지 않는다.
  - 예: `game_state`는 시스템을 조합만 하고 계산 세부 구현은 각 시스템에 둔다.

- 추가 정론 원칙:
  - `KISS`: 복잡한 추상화보다 단순하고 읽기 쉬운 구현 우선.
  - `DRY`: 중복 계산/상수/조건은 공통 함수 또는 테이블로 통합.
  - `YAGNI`: 현재 요구(v1) 없는 구조(멀티플레이, 서버, 과도한 ECS)는 넣지 않는다.
  - `Fail Fast`: 잘못된 입력/상태는 조용히 무시하지 말고 명시적 메시지로 드러낸다.

## 4) 코드 구조 원칙
- 계층 구조를 유지한다:
- `main.lua`: LÖVE 콜백 위임만 담당하는 초경량 엔트리.
- `src/application/*`: 입력 유스케이스, 상태 전이, 게임 틱 오케스트레이션.
- `src/presentation/*`: 렌더링과 HUD 출력.
- `src/*_system.lua`, `src/*_controller.lua`: 도메인 규칙/계산 담당.
- `src/save_system.lua`: 로컬 저장 인프라 담당.
- `src/game_state.lua`: 상태 생성과 저장 스냅샷 조립.
- `src/player_controller.lua`: 포인터 추적 이동, 섭취 반경 계산.
- `src/food_system.lua`: 먹이 스폰/이동/섭취 보상.
- `src/skill_tree_system.lua`: 노드 데이터, 의존성, 해금 로직.
- `src/map_system.lua`: 맵 해금/전환/보상 배율.
- `src/boss_system.lua`: 최종 보스 입장/전투/승리 처리.
- 신규 시스템 추가 시 단일 책임 원칙을 유지하고, 기존 모듈에 기능을 과도하게 누적하지 않는다.

## 5) 게임플레이 설계 규칙
- 입력 피로 최소화: 기본 섭취는 자동 충돌 처리.
- 성장 체감 보장: 초반 20~30분 내 유의미한 파워 상승이 느껴져야 한다.
- 성장 정체 금지: 15분 이상 체감 성장 정체 구간이 나오지 않게 밸런스 조정.
- 스킬트리 우선: 새 기능은 가능하면 스킬트리 노드 효과로 편입.
- 맵은 가속 장치: 상위 맵은 난이도 상승과 함께 성장 효율도 같이 올라야 한다.

## 6) 저장/데이터 안전 규칙
- 자동 저장 기본 주기: 30초.
- 강제 저장 트리거: 맵 전환, 보스 입장 전후, 수동 저장, 게임 종료.
- 저장 실패 시 사용자에게 상태를 HUD에 노출.
- `save.json` 손상 시 `save.bak` 복구를 우선 시도.
- 저장 데이터 스키마 변경 시 `version`을 증가시키고 마이그레이션 경로를 명시.

## 7) 구현 스타일
- 숫자 상수는 `src/constants.lua`로 모은다.
- 매직 넘버를 로직에 직접 박지 않는다.
- 렌더링 코드에서 게임 규칙을 계산하지 않는다. 계산은 시스템 모듈에서 처리.
- 디버그용 메시지는 사용자 HUD를 오염시키지 않게 짧고 상황성 있게 유지.
- 주석은 필요한 이유 중심으로 짧게 작성한다.

## 8) 테스트 체크리스트
- 조작: 포인터 추적 이동과 충돌 섭취가 안정적으로 동작하는지.
- 성장: 스킬 해금 시 보너스가 즉시 반영되는지.
- 진행: 스킬 해금 수에 따라 맵 해금이 정확히 열리는지.
- 보스: 입장 조건/피해 계산/승리 보상이 정상인지.
- 엔딩: 3개 조건 동시 만족 시에만 엔딩 처리되는지.
- 저장: 자동/수동 저장 후 재실행 시 상태가 동일하게 복원되는지.

## 9) 변경 작업 규칙
- 큰 변경은 반드시 아래 순서로 진행:
  1. 영향 시스템 식별
  2. 데이터 계약(저장 스키마/상수) 확인
  3. 로직 수정
  4. HUD/입력 연결
  5. 저장/복구 회귀 확인
- 밸런스 변경은 코드와 함께 변경 이유를 커밋 메시지 또는 문서에 남긴다.
- PR/리뷰 시 체크:
  - 이 변경이 단일 책임을 깨는가?
  - 하드코딩 분기를 데이터 확장으로 바꿀 수 있는가?
  - 테스트 체크리스트에 회귀 위험이 있는가?

## 10) v1 이후 확장 우선순위
- 1순위: 밸런스 테이블 외부화(데이터 드리븐 강화).
- 2순위: 아트/애니메이션/SFX 추가.
- 3순위: 튜토리얼/가이드 UX 개선.
- 4순위: 모바일/패드 대응 검토.
